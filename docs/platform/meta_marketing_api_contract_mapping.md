## Object Creation Order

The offline importer should model platform writes in this order, even while staying dry-run:

1. Resolve `account_id_alias` to a real `act_<AD_ACCOUNT_ID>` only after HITL.
2. Create or resolve Campaign.
3. Create or resolve Ad Set.
4. Upload or resolve creative media, preferably into a Meta Business Creative Asset Management folder, falling back to ad-account media assets when folder access is unavailable or insufficient.
5. Create Ad Creative.
6. Create Ad.
7. Read preview/review result.
8. Pause/activate only after explicit approval.

## Field Mapping

| Manifest/export field | Platform target | Status |
| --- | --- | --- |
| `account_id_alias` | `AdAccount` id such as `act_<id>` | Alias must never be sent directly. Needs resolver and account-ownership proof. |
| `campaign_key` | Campaign `name`, or existing campaign resolver | Safe as local name. Existing campaign lookup is future read-only proof. |
| `objective` | Campaign `objective` enum | Current values `traffic` and `sales` need a canonical enum map, likely `OUTCOME_TRAFFIC` and `OUTCOME_SALES`. |
| `adset_key` | AdSet `name` plus `campaign_id` | Safe as local name. Requires campaign id before write. |
| `placement` | `Targeting.publisher_platforms`, `facebook_positions`, `instagram_positions`, etc. | Current simple values need a placement-spec map, not direct passthrough. |
| `country` | `Targeting.geo_locations` | Current country code is usable input but needs full geo object. |
| `destination_url` / final URL preview | `AdCreative.object_story_spec.link_data.link` plus `url_tags` | Must split URL and UTM tags. Current final preview is evidence only. |
| `utm_*` | `AdCreative.url_tags` or link URL query policy | Needs canonical encoding and duplicate-query handling. |
| `primary_text` | `AdCreative.object_story_spec.link_data.message` | Needs per-format creative rules. |
| `headline` | `AdCreative.object_story_spec.link_data.name` or creative `title` | Needs per-format creative rules. |
| `asset_hash` | Local checksum plus future `AdCreative.image_hash` after image upload | Synthetic hash is not Meta image hash. Requires upload response mapping. |
| video asset metadata | Local checksum plus future `AdCreative.video_id` after video upload | Requires business-folder or `advideos` upload response mapping. |
| future `creative_folder_id` | Business Creative Asset Management folder id | Preferred Meta-native destination for customer-visible creative management; blocked until G3/G4 proof. |
| `post_id_type=existing` / `post_id` | `AdCreative.object_story_id` or source post fields | Requires read-only post ownership and compatibility proof. |
| `operation_intent` | create ad, reuse existing post, or block | Must control the creative path; unsupported intents block. |
| `idempotency_key` | local idempotency ledger | Meta field not identified. Keep local-only unless a safe external id field is proven. |
| `approval_record` | local audit plus future status decision | Do not equate agency approval with Meta ad review. |

## Create Parameter Highlights From SDK

The official autogenerated SDK exposes these relevant write params:

- Campaign: `name`, `objective`, `status`, `daily_budget`, `lifetime_budget`, `bid_strategy`, `promoted_object`, `special_ad_categories`, `start_time`, and `execution_options`.
- Ad Set: `campaign_id`, `name`, `status`, `targeting`, `optimization_goal`, `billing_event`, `destination_type`, `daily_budget`, `lifetime_budget`, `promoted_object`, `start_time`, `end_time`, and `execution_options`.
- Ad Creative: `name`, `object_story_spec`, `object_story_id`, `asset_feed_spec`, `image_hash`, `image_file`, `image_url`, `link_url`, `body`, `title`, `url_tags`, `degrees_of_freedom_spec`, and `execution_options`.
- Ad: `adset_id`, `creative`, `name`, `status`, `tracking_specs`, `conversion_domain`, `execution_options`, and related scheduling/priority params.
- Image upload: `bytes` on `adimages`.
- Video upload: `upload_phase`, `file_size`, `fbuploader_video_file_chunk`, `title`, and `name` on `advideos`.
- Business creative folder: `name`, `description`, and `parent_folder_id` on `creative_folders`.
- Business image upload: `bytes`, `name`, `creative_folder_id`, and placement validation fields on `images`.
- Business video upload: `creative_folder_id`, upload fields, placement validation fields, `title`, and file/chunk fields on `videos`.

## Adapter Implications

- Keep `meta_api_compatibility: not_claimed` until a read-only sandbox proof validates the mapping.
- Add `platform_contract_status: mapped_not_executed` only after tests compare export fields to this mapping.
- Add a canonical enum module before any adapter:
  - objective map;
  - placement map;
  - billing/optimization defaults;
  - status defaults;
  - special ad category policy;
  - CTA policy.
- Add a platform payload builder that emits **draft payloads only** with `execution_options: ["validate_only"]` when a sandbox path is approved.
- Treat image/video upload response IDs and hashes as separate platform lineage, not replacements for source asset hashes.
- Prefer Meta-native asset handoff over local retention: upload originals into a business creative folder or ad-account media asset, then persist only source checksum, Meta hash/id, creative folder id, source-row lineage, and audit status.
- Treat existing post reuse as blocked until ownership, account, page/IG identity, and creative compatibility are proven.

## Hard Blocks Before Code Calls Meta

- No app id, app secret, OAuth token, system-user token, page token, or sandbox token may enter repo files, vault notes, generated artifacts, or logs.
- No customer manifest may be loaded until the data-classification and retention gate is approved.
- No live account may be queried until HITL approves the account, token scope, target endpoint, and read-only command.
- No upload may run until asset retention/deletion, hashing, redaction, and operator approval are documented.
- No zero-retention claim may be made for real customer assets until G1/G4 define the transient cache, deletion receipt, upload-response lineage, and failure retry TTL.
- No mutation may run until dry-run diff, validate-only result, preview, rollback/pause path, and two-key Platform/Security approval exist.
